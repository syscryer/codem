# Refactor Guidelines

## 核心原则

- 先拆职责，再谈抽象
- 每次重构优先让主入口文件明显变薄
- 结构拆分不能以行为回归为代价

## 推荐顺序

1. 先抽类型和常量
2. 再抽纯 helper
3. 再抽独立组件
4. 再抽 hook
5. 最后才考虑 service / domain 层

## 风险信号

- 一个 hook 需要知道另一个 hook 的内部实现
- 组件 props 越拆越长
- 同一状态被多个层级重复保存
- 菜单/弹层状态被提升到全局但只在局部使用

## 实操要求

- 每个阶段至少跑一次 `npm run typecheck`
- 结构改动大时，优先小步提交
- 新增文件命名要表达职责，避免 `utils.ts` 这类模糊名字
