# 会话预览动作收口设计

## 背景

当前会话消息中的文件相关入口存在语义混用：

- 代码变更摘要行出现 `打开` 按钮，与展开卡片中的右侧入口重复
- 代码文件本应以“审查改动”为主，但部分入口表现为“预览/打开”
- 文档和结果类文件缺少独立、明确的快捷打开入口

这会导致用户对“打开”和“审查”的区别不明确，也容易出现入口冗余。

## 目标

收口会话内文件动作的交互语义：

- 代码类文件统一走 `审查`
- 文档/结果类文件统一走 `打开`
- 去掉重复入口，保持消息摘要层简洁

## 交互方案

### 1. 代码变更卡

适用范围：

- `Edit`
- `Write`
- `NotebookEdit`
- 其他会生成代码 diff 的工具预览

行为：

- 摘要行不再展示 `打开`
- 摘要行仅展示文件名、增删行数、展开状态
- 展开后的代码卡片保留现有右侧入口
- 该入口文案统一为 `审查` 或 `在右侧审查`
- 点击后进入右侧工作台 `审查` 页，而不是文件浏览页

### 2. 产出文件区块

适用范围：

- Markdown
- HTML
- TXT
- JSON
- CSV
- YAML
- 其他更偏“结果查看”的文件

行为：

- 在消息底部新增 `产出文件` 区块
- 每个文件一行
- 左键主按钮为 `打开`
- 文本类文件左键整卡进入右侧工作台 `文件` 页，展示文件正文
- Office / PDF 类文件左键整卡使用系统默认应用打开
- 会话产出文件进入右侧预览时，不主动展开文件浏览栏
- 如果文件浏览栏是用户此前手动打开的，则保持打开，不因为会话产出预览而被关闭

### 3. 产出文件右键菜单

右键菜单项：

- `打开`
- `在文件浏览器打开`
- `复制路径`

其中：

- `在文件浏览器打开` 在 Windows 下使用资源管理器定位并选中文件
- 不只是打开目录

## 视觉层级

消息内的层级保持为：

1. 外层摘要：轻量信息，不承载主动作
2. 展开后的代码卡片：承载 `审查`
3. 底部产出文件区块：承载 `打开`

这样可避免把整个消息顶部做成操作条，也能让“代码审查”和“文档打开”形成稳定心智。

## 预期结果

- 代码文件不再出现“打开/预览/审查”混杂
- 文档文件有明确的快捷打开入口
- 右侧工作台的“审查”和“文件浏览”职责更清晰
- 会话产出预览默认聚焦右侧预览内容，不额外打扰文件浏览状态
- 会话消息的视觉噪音减少

## 非目标

- 本次不重做现有右侧工作台结构
- 本次不改动 Git 审查能力本身
- 本次不扩展到所有工具卡，只聚焦文件相关入口
