# CodeM Agent Guide

本仓库当前使用轻量 `.trellis` 结构管理开发规范。

## 阅读顺序

开始较大改动前，建议按下面顺序建立上下文：

1. `README.md`
2. `.trellis/workflow.md`
3. `.trellis/spec/frontend/index.md`
4. `.trellis/spec/backend/index.md`
5. `.trellis/spec/guides/index.md`

## 当前约定

- 默认所有对用户的回复、过程汇报、方案说明、提交说明使用中文；除非用户明确要求，否则不要改用英文。
- 所有新功能在实现前都必须先和用户讨论清楚需求边界、成功标准、交互预期、数据流、安全/隐私影响、兼容策略和验收方式；不允许在关键需求不明确时直接边猜边写。
- 如果新功能涉及多个可能方向，应先给出 2-3 个可选方案、取舍和推荐方案，等用户确认后再进入设计或编码。
- frontend 代码范围：`src/**`
- backend 代码范围：`server/**`
- 类型与常量优先集中到 `src/types.ts`、`src/constants.ts`
- 纯 helper 优先放 `src/lib/**`
- 共享行为优先放 `src/hooks/**`
- 页面结构块优先放 `src/components/**`
- 修改前端或后端代码后，如果判断当前运行中的开发服务需要刷新生效，应主动重启项目服务，并在回复里说明已重启。
- 当用户说“启动项目”“启动起来”等未特别限定端别的指令时，默认启动桌面开发模式 `npm run desktop:dev`；只有在用户明确说明只启动 Web/后端、只看浏览器页面，或当前任务明确不需要桌面壳时，才改为启动 `npm run dev`。
- 仅修改 Web 版相关代码时，不要顺手构建或重启桌面版；仅在修改桌面壳、Tauri 配置、窗口材质、桌面专属样式或用户明确要求查看桌面版时，才主动构建并重启桌面版。
- 修改哪一端就重启哪一端：Web 端改动只重启 Web/后端开发服务；桌面端、Tauri、打包资源、桌面专属样式或窗口配置改动，应重启桌面版，用户要求查看安装包效果时再打包后重启。
- 需要推送代码时，默认先推送到 Gitee 远端 `gitee` 的 `main` 分支，再推送到 GitHub 远端 `origin` 的 `main` 分支；除非用户明确要求只推某一个远端。
- 修复问题时优先定位并修正真实数据来源或真实流程，尽可能不要用兜底逻辑掩盖问题；只有在必须兼容旧数据、外部异常或不可控输入时才加兜底，并说明原因。
- 更新桌面图标时，需要同步 `icon.png`、`public/icon.png`、`src-tauri/icons/icon.png` 和 `src-tauri/icons/icon.ico`；Windows/Tauri 会读取 `.ico` 的第一帧作为默认窗口图标，第一帧必须保持 `256x256`，并保留 `16/20/24/30/32/36/40/48/64/128/256` 等小尺寸帧，避免任务栏或开发模式图标发糊。
- 如果开发模式图标没有变化，先确认 `src-tauri/target/debug/codem.exe` 是否已重新生成；必要时停止桌面 dev 壳后执行 `cargo clean -p codem`，再启动 `npm run desktop:dev`，不要只反复改图片文件或重新打包。
- 遇到成熟软件已有的复杂交互、状态机或布局算法（例如 Git 图、Diff、编辑器、终端、文件树、虚拟列表等），不要长时间只靠截图和猜测规则硬调；应优先查阅可信开源项目源码、文档或测试用例，参考其数据模型、边界条件和渲染策略。
- 本项目开源，允许在许可证兼容（如 MIT、Apache-2.0、BSD 等）的前提下借鉴成熟开源实现；不能无脑整段搬运，需适配本项目的数据结构、主题变量、交互风格和性能要求，并在代码注释或相关文档中保留来源/许可证说明。
- 对看似只是 UI、实际包含图形拓扑或状态流转的问题，优先用开源参考实现校准方案，并补充能覆盖关键拓扑/状态的回归测试，再做视觉微调。
- 前端页面、卡片、弹窗、工具栏、设置项、图表、列表、开关、输入框等视觉元素需要保持统一风格，避免新旧样式割裂。
- 涉及外观、主题、设置、工作台、会话管理、MCP、插件等界面时，不能只改单个区域，需同步检查同类组件是否仍使用旧样式或硬编码颜色。
- 主题切换必须尽量全局联动，不能只切容器背景；统计图、卡片、构建/预览区、开关、按钮、输入框、hover、边框、滚动条等都应跟随主题变量变化。
- 新增样式优先复用现有主题变量和通用样式类，尽量避免直接写死浅色或深色值；确需新增颜色时，优先补充到统一变量体系。
- 同一类交互控件在不同页面要尽量保持一致的圆角、边框、背景、悬浮态、激活态和禁用态，不要出现某页是原生控件、某页是自定义控件但风格不一致的情况。
- 下拉选择器优先复用统一的自定义下拉样式和现有菜单/弹层体系，非必要不要直接使用原生 `select` 默认展开样式。
- 修改设置页、弹出层、下拉、菜单、卡片布局时，要同时关注桌面端和 Web 端观感，避免只在局部页面可用、整体风格却不统一。
- 涉及工作台、文件树、审查区、Diff 预览、代码预览、会话切换、标签切换、大文件展示等场景时，必须默认考虑性能影响，不能只实现功能后再补救卡顿问题。
- 对上述高频交互，优先避免不必要的组件卸载重建、重复请求、整树重渲染和整文件全量渲染；能缓存就缓存，能增量就增量，能虚拟滚动就不要一次性挂载大量 DOM。
- 修改预览、Diff、高亮、文件树、列表这类容易放大性能问题的区域时，要同时评估大文件、长列表、多标签切换和频繁切换场景，避免“普通文件正常、大文件明显卡顿”的实现。
- 如果发现当前问题更像是低价值细节、边际收益很小的视觉微调，或继续深挖需要明显扩大改动范围但对主流程帮助有限，应主动提醒用户这件事可能不值得继续死扣，给出“先收住 / 接受当前结果 / 后续再统一处理”等建议，避免双方在意义不大的问题上投入过多时间。

## 聊天输入、@文件与附件需求约束

- 这块属于核心会话体验，后续实现前必须先把需求讨论清楚，不要直接边猜边写代码。
- 详细需求草案见 `openspec/chat-input-content-blocks-and-attachments.md`；实现前必须先确认该文档的阶段范围和验收标准。
- 落地原则是“协议中立，功能轻量”：内部模型不能绑死 Claude，但第一阶段只做高频闭环，不引入 PDF/DOCX 深度解析、复杂拖拽、完整文件预览器等低频重功能。
- 目标是往成熟产品品质靠近，不做只解决单一路径的妥协实现；文本、图片、项目文件引用、上传附件、后续 PDF/Word 等输入都应逐步收口到统一的 `contentBlocks` 数据模型。
- `/api/claude/run` 可以继续兼容旧的 `prompt + attachments` 请求，但 bridge 内部应优先 normalize 成统一 content blocks，再写入 Claude stdin。
- `@文件` 需要先明确语义，推荐默认使用智能策略：小文本、代码、Markdown 文件可以内联为文本块；大文件、图片、PDF/Word、二进制文件默认作为文件引用块，让模型使用合适工具读取。
- 上传附件需要和 `@文件` 共用一套附件模型和 UI 展示，不要让图片、普通文件、项目文件引用分别长出三套独立逻辑。
- 图片需要同时支持多模态模型的 image block，以及非多模态模型可用的本地路径兜底；提示词不得诱导模型用 `Read`、`Grep` 等文本工具读取图片。
- 历史记录、debug events、raw events、trace 日志都不能保存或展示 base64、大文件全文或敏感文件内容；trace 只记录类型、数量、大小等摘要。
- 发送队列、运行中 guide、重试、恢复会话等桥接路径都必须保留附件语义，不能只保证普通发送路径可用。
- UI 上应让用户清楚看到每个输入块的状态，例如已内联、仅引用、图片、上传文件、项目文件引用、过大未内联等。
- 实现前应补充测试覆盖 content blocks 归一化、历史脱敏、trace 摘要、队列附件不丢、guide 附件隔离、非多模态路径兜底。

## 当前阶段说明

仓库目前只启用了轻量规范骨架：

- 有规范文档
- 有任务目录
- 没有自动化脚本体系
- 没有 developer workspace / session record 流程

如果后续团队协作规模扩大，可以在当前骨架上继续补全。
